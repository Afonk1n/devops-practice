## Скрипт для собеса DevOps (под твою вакансию)

Ниже — заготовка речи + ключевые пункты, которые ты уже можешь честно сказать, опираясь на сделанное.

### 1. Кратко о себе

- «Я сейчас на 4 курсе в универе, базовое ИТ-образование: немного программирование, немного администрирование.
  Конкретно DevOps начал целенаправленно трогать недавно, но за последний день/две очень плотно прошёл Docker, Kubernetes, GitLab/GitHub CI-подобный сценарий и базовое окружение под веб-приложения.»

Если спросят про опыт:
- «Пока больше учебные и pet-проекты, без продакшн-ответственности. Зато есть хорошее понимание базовой цепочки: от кода до запуска в Kubernetes, и готов быстро доучиваться под задачи команды.»

### 2. Docker и контейнеризация

- «Умею читать и писать простые Dockerfile. В учебном проекте у меня два сценария:
  - Python-сервис (`simple-app`), который упакован в образ на базе `python:3.12-slim` с простым HTTP-сервером;
  - Nginx-образ (`devops-nginx`) на базе `nginx:alpine` с кастомной стартовой страницей и своим `nginx.conf`.»
- «Локально собирал образы через `docker build` и запускал контейнеры с пробросом портов:
  - `docker run --rm -p 8000:8000 ci-demo-app` — Python-сервис;
  - `docker run --rm -p 8080:80 devops-nginx` — Nginx со своей страницей.
  Понимаю разницу между образом и контейнером, что делает `EXPOSE`, `CMD` и как работает проброс портов `-p host:container`.»
- «Через Docker Desktop GUI смотрю образы и контейнеры, порты и логи — удобно для быстрой диагностики без CLI.»

### 3. Git и CI (GitHub Actions)

- «С Git работаю на уровне повседневных задач: `status`, `add`, `commit`, `branch`, `checkout -b`, `push`. Привык заводить отдельную ветку под фичу и пушить её на GitHub.»
- «Сделал простой CI на GitHub Actions: при каждом push в `master` в репозитории запускается workflow:
  - чек-аутит код;
  - на чистой Ubuntu-среде собирает Docker-образ `ci-demo-app` из каталога `simple-app` с помощью официального `docker/build-push-action`.
  Это по сути автоматизация того, что я делаю локально руками через `docker build`.»
- «Понимаю базовую структуру CI-пайплайна: `on` — события, `jobs` — независимые задачи, `steps` — конкретные шаги в jobе. Дальше на такой пайплайн уже можно навешивать тесты и деплой в кластер.»

### 4. Kubernetes (Pods, Deployments, Services, Ingress, ConfigMaps, Secrets)

- «Разворачивал простое приложение в Kubernetes-кластере Docker Desktop:
  - написал Deployment `demo-nginx` с двумя репликами Pod-ов на `nginx:alpine`;
  - сделал Service `demo-nginx` типа `ClusterIP`, который балансирует трафик по этим Pod-ам;
  - описал Ingress, который по хосту `demo.local` направляет HTTP-трафик в этот Service.»
- «Работаю с базовыми объектами:
  - `Pod` — минимальная единица, где крутится контейнер;
  - `Deployment` — следит, чтобы нужное число Pod-ов всегда было в статусе Ready;
  - `Service` — стабильная точка входа по имени, балансирует между Pod-ами;
  - `Ingress` — входной HTTP-трафик в кластер;
  - `ConfigMap`/`Secret` — передача конфигурации и секретов в env-переменные контейнера.»
- «Использовал `kubectl apply/get/describe/logs/port-forward`:
  - через `kubectl describe pod` смотрю, какой образ, порты, env-переменные из ConfigMap/Secret и состояние контейнера;
  - через `kubectl logs` смотрю логи конкретного Pod-а;
  - через `kubectl port-forward svc/demo-nginx 8081:80` подключался к сервису из браузера (`http://localhost:8081` → Service → Pod).»

### 5. Nginx, HTTP и окружение для веб-приложений

- «Понимаю общую схему для веб-приложений, в том числе Laravel:
  - пользователь делает HTTP-запрос;
  - Nginx принимает его на 80/443 порту, сам раздаёт статику;
  - динамические запросы прокидывает дальше — к приложению (PHP-FPM/Octane для Laravel);
  - приложение ходит в базу/Redis и формирует ответ, который Nginx возвращает клиенту.»
- «Собирал свой Nginx-образ с конфигом:
  - `server { listen 80; root /usr/share/nginx/html; ... }`;
  - отдельные `location` для статики и основного приложения;
  - понимая идею `try_files` и того, что все запросы можно свести к `index.php` или `index.html`, где уже роутит фреймворк.»
- «Экстраполяция на Laravel: конфигурация окружения через `.env` и env-переменные (`APP_ENV`, `DB_*`, `REDIS_*`, очереди и т.п.), которые в Kubernetes удобно хранить в ConfigMap и Secret и прокидывать в Pod.»

### 6. Мониторинг и логирование (Prometheus, Grafana, логи)

- «На уровне концепций понимаю стек мониторинга:
  - Prometheus регулярно ходит на `/metrics` сервисов и кластера, собирает числовые метрики (нагрузка, ошибки, задержки);
  - Grafana строит дашборды и позволяет быстро увидеть состояние сервисов;
  - алерты настраиваются через Alertmanager (например, по росту error rate или задержки).»
- «Важные для меня метрики: доступность сервиса, частота ошибок, время ответа, загрузка CPU/памяти, состояние базы и очередей.»
- «Для оперативной диагностики привык смотреть логи:
  - `docker logs` для отдельных контейнеров;
  - `kubectl logs` для Pod-ов в Kubernetes;
  - а в Docker Desktop GUI — логи контейнеров и Kubernetes Pod-ов через интерфейс.»

### 7. Как ответить на общие вопросы «почему DevOps» и «чего хочешь»

- «Мне интересно как раз стык разработки и эксплуатации: понимать, как код живёт в реальном окружении, как его запускать, обновлять и мониторить.
  Сейчас я на уровне junior/стажёра, хочу под руководством более опытных ребят углубиться в Kubernetes, CI/CD и инфраструктуру вокруг Laravel-приложений, чтобы со временем перейти в сторону SRE/платформенной инженерии.»
- «Из ближайшего хочу прокачать:
  - более сложные пайплайны CI/CD (деплой в кластер, миграции, health-check-и);
  - Helm-чарты для типовых сервисов;
  - мониторинг/логирование в боевых условиях.»

### 8. Примеры вопросов, к которым ты готов

- «Расскажи, как ты деплоил своё приложение в Kubernetes?»  
  → отвечаешь про `k8s-demo`: Deployment (2 реплики Nginx), Service, Ingress, ConfigMap/Secret, `kubectl apply`, `kubectl port-forward`, `kubectl describe/logs`.

- «Что такое Deployment и чем он отличается от Pod?»  
  → Pod — конкретный инстанс контейнеров; Deployment — контроллер, который управляет набором Pod-ов (сколько их должно быть, как обновлять, как откатываться).

- «Как бы ты организовал CI/CD для Laravel-приложения?»  
  → на основе твоего опыта с GitHub Actions:
  - шаги: checkout → тесты → сборка Docker-образа → пуш в registry → деплой в Kubernetes (kubectl/Helm/ArgoCD);
  - проговорить, что env/секреты через ConfigMap/Secret, миграции через отдельный job/step.

- «Как ты диагностируешь проблему, если сервис в Kubernetes не отвечает?»  
  → цепочка:
  - `kubectl get pods`, `kubectl describe pod`, `kubectl logs`;
  - проверка Service (`kubectl get svc`, `kubectl describe svc`);
  - при необходимости `kubectl port-forward` и локальный curl/браузер.

- «Что знаешь про мониторинг и алертинг?»  
  → отвечаешь про Prometheus/Grafana, метрики, Alertmanager, примеры алертов.
